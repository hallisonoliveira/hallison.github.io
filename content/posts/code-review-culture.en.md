Code reviews can feel like gatekeeping or nitpicking. But they're actually your team's best tool for collective code ownership and knowledge sharing.

## What Reviews Aren't

- **A power trip**: "My way or the highway"
- **Grammar checking**: Spelling in comments isn't the point
- **Nitpicking**: Don't block on unused variables (linters do that)
- **A test suite**: Reviews aren't about finding bugs

## What Reviews Are

- **Knowledge sharing**: Spread understanding across the team
- **Collective ownership**: Everyone can maintain any code
- **Design feedback**: Is this approach sustainable?
- **Learning opportunity**: Both ways

## How to Conduct Them

**As the reviewer:**
- Ask questions instead of demanding: "What's the reason for this?" instead of "This is wrong"
- Praise good decisions
- Distinguish between "must fix" and "consider"
- Be specific: "Line 42, this could be simplified to..."

**As the author:**
- Respond to comments, not with defensiveness, but with explanation
- You don't have to agree, but explain why
- Use reviews to improve your code

## Red Flags

- Reviews that take hours to get feedback
- Comments like "just do it my way"
- Using reviews to avoid pairing/discussing
- Reviewing code you don't understand (ask questions!)

## Conclusion

Code reviews are conversations, not judgments. Done right, they make everyone better.
